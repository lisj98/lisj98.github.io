1、快速开始

要快速启动和运行 MyBatis 生成器（MBG），需要遵循以下步骤（此处为命令行方式生成，还可使用 Maven 插件或 Java 代码生成，见下方 2、运行 MBG）：
1、创建并编写一个配置文件
2、将文件保存在一些方便的位置(如 \temp\generatorConfig.xml)
3、在命令行中使用如下命令运行MBG：

这将告诉 MBG 使用自定义的配置文件运行。它还会告诉 MBG 用相同的名称覆盖任何现有的 Java 文件。如果您想保存任何现有文件，那么可以省略 -overwrite 参数。如果有冲突，MBG 将以唯一的名称保存新生成的文件(例如 MyClass.java.1)。

如此便生成了所需的代码。

根据配置的不同，MBG 生成的代码也会不同，这是通过在配置元素上指定 targetRuntime 属性（generatorConfig.xml 中）来控制的。

targetRuntime 属性有以下四种选择：

  • MyBatis3DynamicSql（默认值）

    • 生成 Java 代码
    • 不生成 XML 文件，只生成注解
    • 生成的实体类对象是扁平的——没有单独的主键对象（即只有一个 Record 类）
    • 生成的代码依赖于 MyBatis 动态 SQL 库
    • 生成的代码相对较少
    • 生成的代码在查询条件构造方面具有极大的灵活性

    配置示例：

  • MyBatis3Kotlin（省略，见官网）

  • MyBatis3（MBG 1.3.6版本之前大多使用该属性）

    • 生成 Java 代码
    • 生成 MyBatis3 兼容的 XML 配置文件或者不带配置文件的注解（即既能生成 XML 文件，又能生成注解，通过 <javaClientGenerator type=”xxx”/> 指定）
    • 生成的实体类对象可能具有单独的主键对象或者带有 BLOB 字段的单独对象（即有一个 Record 类、一个主键类、一个 Record With BLOBs 类，最后会合并为一个实体类展现）
    • 生成的代码没有外部依赖项
    • 生成的代码量非常大
    • 生成的代码查询条件构造能力有限，很难扩展

    配置示例：

  • MyBatis3Simple（MyBatis3 简化版）

    • 生成Java代码
    • 生成 MyBatis3 兼容的 XML 配置文件或者不带配置文件的注解（即既能生成 XML 文件，又能生成注解）
    • 生成的实体类对象是扁平的——没有单独的主键对象（即只有一个 Record 类）
    • 生成的代码没有外部依赖项
    • 生成的代码相对较少
    • 不生成“by example”或“selective”方法，即生成的代码不包括用于动态查询构造的方法，很难扩展

参数名 说明
configurationFile MBG 配置文件的位置，默认值为 ${basedir}/src/main/resources/generatorConfig.xml
contexts 和 tableNames 使用方法一致
jdbcDriver 数据库驱动
jdbcPassword 数据库密码
jdbcURL 数据库地址
jdbcUserId 数据库用户名
outputDirectory MBG 生成的新文件被放置的目录，当 targetProject 被指定为“MAVEN”（大小写敏感）时用到，默认值为 ${project.build.directory}/generated-sources/mybatis-generator
overwrite 默认值为 false。①如果为真，那么如果现有的 Java 文件与生成的文件具有相同的名称，现有的 Java 文件将被覆盖；②如果没有指定或者为假，并且已经存在与生成的文件同名的 Java 文件，那么 MBG 将新生成唯一的名称，与旧文件写入同一目录；③MBG 总是会合并和覆盖旧 XML 文件
sqlScript 生成代码之前要运行的 SQL 脚本文件的位置，值可以指定为文件系统中的一个位置，如果前缀为“classpath”，则可以指定构建类路径上的一个位置。①如果为null，则不运行任何脚本；②如果不为空，则还必须提供 jdbcDriver、jdbcURL等
tableNames 指定哪些表需要生成代码（注意需要和 generatorConfig.xml 中 table 标签指定的表名一致，即必须要在该文件中找到这个表名，找不到或者名字错误均不生成代码），不指定则为 generatorConfig.xml 中所有 table 标签指定的表生成代码
verbose 如果为真，则 MBG 将把进度消息写入构建日志
includeCompileDependencies 如果为真，那么范围为”compile”， “provided”和”system”的依赖项将被添加到生成器的类路径中
includeAllDependencies 如果为真，那么任何作用域的依赖项都将添加到生成器的类路径中

targetProject 属性备注：
当使用 Maven 运行时，生成器对配置的 targetProject 属性的解释是不同的。如果设置为特殊值”MAVEN”(区分大小写)，则 targetProject 将被设置为插件的输出目录（outputDirectory），如果该目录不存在，则会创建该目录。如果没有设置为”MAVEN”，那么 targetProject 将被 MBG 解释为常规目录，它必须被设置为一个已经存在的目录。

3、生成代码过程详解

MBG 根据 targetRuntime 属性的配置生成不同的对象。具体有 MyBatis3DynamicSql（推荐）、MyBatis3 或 MyBatis3Simple（传统）这三种。

3.1、使用 MyBatis Dynamic SQL 生成对象

即使用 MyBatis3DynamicSql。
这种情况下，对于每一个表，MBG 生成三个类：

  • 一个 Record 类代表数据库表的一行记录（即我们通常意义上的实体类）
  • 一个 Support 类包括表和所有列的定义（主要用于 Mapper 接口中拼接 where 子句）
  • 一个 Mapper 接口带有所有通用 SQL 方法

关于生成的代码，还有几点要说明：

  • 没有 XML 文件生成
  • 没有单独的"Example"类生成
  • 没有单独的处理“BLOB”字段的方法
  • 生成的实体类总是采用“flat”模式，即只生成一个 Record 类，不含主键类和 Record With BLOB 类

3.1.1、Support 类的形式

每一个表都有一个”Support”类，这个类包括表和所有列的定义。这些定义通常用于 Mapper 中某些 SQL 语句的输入——最常用于 where 子句的输入。

3.1.5、新插件

因为“*ByExample”方法已经被移除了，因此通过 XML 文件中的 <table> 元素中的属性（如 <enableInsert>，<enableSelectByExample> 等）来控制生成的方法已经没有意义了，这时候可以使用 MBG 提供的新插件来控制：

  • org.mybatis.generator.plugins.dsql.DisableDeletePlugin：不生成删除方法
  • org.mybatis.generator.plugins.dsql.DisableInsertPlugin：不生成插入方法
  • org.mybatis.generator.plugins.dsql.DisableUpdatePlugin：不生成更新方法
  • org.mybatis.generator.plugins.dsql.ReadOnlyPlugin：：不生成所有删除、插入、更新方法

3.2、使用传统方式生成对象

即使用 MyBatis3 或 MyBatis3Simple。

  • 一个 Mapper 接口
  • 一个 Example 类

3.2.4、Example 类的使用

Example 类指定如何构建动态 where 子句，表中的每个非 BLOB 列都可以有选择地包含在 where 子句中。
Example 类包含一个名为 Criteria 的静态内部类，该类保存了将在 where 子句中组合在一起的条件列表。使用不同的 Criteria 类集可以生成几乎无限种 where 子句类型。
具体来说就是构建 where 子句时需要调用 GeneratedCriteria 内部类中的 addxxx 方法，而每调用一次都会将之后 where 子句中要用到的 condition 和 value 加入到 criteria 对象列表中，最后解析 xml 文件时就可以获取到对应的 condition 和 value 了，也就完成了 where 子句的构建。

Criteria 对象可以使用 createCriteria 方法或 Example 类中的 or 方法创建。当使用 createCriteria 方法创建第一个 Criteria 对象时，它会被自动添加到 Criteria 对象列表中——这使得编写一个简单的 where 子句或组合多个子句变得很容易。当使用 or 方法时，Criteria 对象将被添加到所有实例列表中。
建议使用 or 方法来创建 Criteria 对象，这种方法可以使代码更具可读性。

4.9、<commentGenerator>

作用：commentGenerator，顾名思义，为注释生成器，即用来自定义注释。默认实现为org.mybatis.generator.internal.DefaultCommentGenerator。可以用来为字段添加注解，如用到 Swagger 时添加 @ApiModelProperty。
属性列表：

属性名 是否必须 说明
type N 自定义的注释生成器的类型，必须实现org.mybatis.generator.api.CommentGenerator接口，并且要有一个公共的默认构造函数

子元素<property>中属性：

属性名 是否必须 说明
suppressAllComments N 此属性用于指定 MBG 是否将在生成的代码中包含任何注释。该属性支持以下值：①为 true 时任何生成的元素都不会添加注释；②为 false 或不指定时，所有生成的元素都会添加注释。默认值为 false
suppressDate N 此属性用于指定 MBG 是否将在生成的注释中包含生成时间戳。该属性支持以下值：①为 true 时生成的注释中不会添加时间戳；②为 false 或不指定时，所有生成的注释将包括生成元素时的时间戳。默认值为 false
addRemarkComments N 此属性用于指定 MBG 是否将在生成的注释中包括数据库中表和列的备注。该属性支持以下值：①为 true 时，表和列的备注将被添加到生成的注释中；②为 false 或未指定时，所有生成的注释在生成元素时将不包括表和列的备注。默认值为 false。如果 suppressAllComments 选项为 true，则该选项将被忽略
dateFormat N 在将日期写入生成的注释时使用的日期格式字符串。这个字符串将被用来构造一个java.text.SimpleDateFormat对象，该对象的任何有效格式字符串都可以在这里指定。默认情况下，日期字符串将来自java.util.Date上的 toString() 方法。如果 suppressAllComments 选项或者 suppressDate 选项为 true，该选项将被忽略

4.20、<table>

作用：用于配置需要生成代码的数据库表信息。

属性列表：

属性名 是否必须 说明
tableName Y 表名
schema N 数据库模式
catalog N 数据库目录
alias N 为表和列起别名
domainObjectName N 指定生成的实体类名称
mapperName N 指定 Mapper 接口和映射文件名称
sqlProviderName N SQL 提供程序类的名称
enableInsert、enableSelectByPrimaryKey、enableSelectByExample、enableUpdateByPrimaryKey、enableDeleteByPrimaryKey、enableDeleteByExample、enableCountByExample、enableUpdateByExample、selectByPrimaryKeyQueryId、selectByExampleQueryId N 指定是否需要生成这些方法
modelType N 指定如何生成类，是只生成一个主键类，还是需要生成 BLOB 类、 Record 类，取值有：①conditional；②flat；③hierarchical（参考defaultModelType）
escapeWildcards N 表示在搜索列时，模式和表名中的 SQL 通配符(‘_’和’%’)是否应该转义。如果模式或表名包含一个 SQL 通配符(例如，如果表名是 MY_TABLE，一些驱动程序要求将下划线字符转义)，那么这是必需的
delimitIdentifiers N 指定是否需要在生成的 SQL 中为标识符（类似于 Java 关键字）添加分隔符
delimitAllColumns N 指定是否需要在生成的 SQL 中为所有列添加分隔符

子元素有：

  • <property> (0..N)
  • <generatedKey> (0 or 1)
  • <domainObjectRenamingRule> (0 or 1)
  • <columnRenamingRule> (0 or 1)
  • <columnOverride> (0..N)
  • <ignoreColumn> (0..N)

子元素<property>中还有如下属性：

属性名 是否必须 说明
constructorBased N 指定是否为类生成构造函数。若该属性被指定，则对象将通过构造函数构建，而不通过 setter 方法。如果”immutable”属性设置为”true”，则该属性将被忽略(并强制为”true”)
dynamicSqlSupportClassName N MyBatis 动态 SQL Support 类的名称
dynamicSqlTableObjectName N MyBatis 动态 SQL Support 类中生成的内部类所使用的名称
ignoreQualifiersAtRuntime N 如果为 true，则 MBG 不会将模式或目录添加到生成的 SQL 中的表名前面。如果你在多个模式中有相同名称的表，这很有用——你可以使用 MBG 基于一个模式中的表生成对象，但不包括运行时的模式
immutable N 用于选择 MBG 是否会生成不可变的实体类——这意味着类将没有“setter”方法，构造函数将接受类中每个字段的值。如果为真，这将强制使用参数化构造函数构建实体类，而不管“constructorBased”属性的值是多少
modelOnly N 用于配置是否只生成实体类，不生成 Mapper 接口和映射文件
rootClass N 为所有生成的实体类指定一个父类
rootInterface N 为所有生成的 Mapper 接口指定一个父接口
runtimeCatalog N 如果为该属性指定一个值，那么 MBG 将使用该值作为生成的 SQL 中的目录，而不是原数据库目录。如果你希望针对一个目录生成代码，但又希望在运行时使用不同的目录，那么这很有用
runtimeSchema N 如果为该属性指定一个值，那么 MBG 将使用该值作为生成的 SQL 中的模式，而不是原数据库模式。如果你希望针对一个模式生成代码，但又希望在运行时使用不同的模式，那么这很有用
runtimeTableName N 如果为该属性指定一个值，那么 MBG 将使用该值作为生成的 SQL 中的表名，而不是原数据库表名
selectAllOrderByClause N 该属性值会追加到 selectAll 方法后的 SQL 中，直接与 order by 拼接后添加到 SQL 末尾
trimStrings N 若数据库返回的字段中有空白部分，此属性可以对其进行特别填充
useActualColumnNames N 指定是否用真实的列名而不加修改
useColumnIndexes N 如果为 true，MBG 生成 resultMap 时会使用列的索引，而不是结果中列名的顺序
useCompoundPropertyNames N 如果为 true，MBG 生成属性名的时候会将列名和列备注结合起来
